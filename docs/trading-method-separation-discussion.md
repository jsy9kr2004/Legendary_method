> 다음 단계: 사용자 자기 인식 확인 (어느 매매법) → 통설 자료 수집 → 라벨링 백테스트.

## 0. 배경

- 사용자(Zeta) 매매 시간대: 09:10 ~ 11:00 (한국 증시 주도주 단타)
- 사용자 자기 인식: 주도주 매매 초보. 매매법 종류 구분이 아직 명확하지 않음
- 현 시스템 상태: `Buy.Score` STRONG/WATCH/NEUTRAL/AVOID 등급 + `Exit.Triggers` A/P/E 트리거 OR 조건. 매매법 구분 없음

## 1. 사용자 통찰 (이 토론의 출발점)

> "주도주 매매에는 다양한 종류가 있는 것 같더라(돌파 매매, 눌림 매매 등). 그런데 시스템이 STRONG / WATCH 로 표현해주는데, 모든 매매법이 뒤죽박죽 섞이는 게 아닐까? 돌파 매매시 보는 STRONG 관점과 눌림 매매시 보는 STRONG 관점이 따로 있을 텐데, 매매법에 맞는 시그널 보고 켜주고 매수하고 그 매매법에 맞는 청산 시그널 보고 파는 게 정석 아닐까? 지금 매수/매도 시그널 혼합돼 있어서 보유 기간 짧고 수익률·승률 안 나오는 거 아닐까?"

→ **이 가설이 본 토론의 핵심.** 데이터 검증 + 통설 검증 후 결론.

## 2. 통설 평가 (Claude 답변 정리)

### (a) 매매법별 분리 필요성 — 한국 단타 통설상 맞음

| 매매법 | 진입 | 손절 | 익절 | 보유시간 |
|---|---|---|---|---|
| **돌파 매매** | 박스/전고점 돌파 + 거래량 동반 | 돌파선 이탈 | 트레일링 (추세 끝까지) | 길어짐 |
| **눌림 매매** | 1차 상승 후 5MA/VWAP 지지 + 거래량 감소→재유입 + 망치형 | 5MA 이탈 또는 직전 저점 | 직전 고점에서 분할 익절 | 짧음 |

진입·손절·익절·보유시간 룰이 모두 다른 별개 시스템. 한 점수에 섞으면 양쪽 다 어중간해짐.

### (b) 현 `Buy.Score` 는 사실 "돌파도 눌림도 아닌 어중간" 상태

- VP / Accel / Candle / Position / Divergence / VWAP / 5-20MA = 모멘텀+위치 지표 짬뽕
- R14k (일중 최고점 거리 페널티) / R14l (횡보 정점 페널티) / R14f (BB 상한 돌파 페널티) = **정점 회피 페널티**
- → "강한 돌파 상태일수록 점수 감점" → 사실상 **돌파 매매와 반대 방향**
- 결과: 어정쩡한 중간 모멘텀에서만 STRONG. 돌파 추종도 못 잡고, 눌림 진입 시점도 못 잡음
- → **사용자 가설 ("보유 짧고 수익률·승률 안 나옴") 과 정합**

### (c) 즉시 시스템 분리는 위험 — ritual 따라야 함

메모리 `feedback_system_tuning_ritual` 기준:

1. 한국 단타 통설 검색 (돌파/눌림 정량 정의 — i-whale / namu.wiki / 단타 책)
2. 현재 tick_log 데이터로 "분리했을 때 vs 현 상태" 백테스트
3. 두 단계 모두 통과 후만 도입

## 3. 옵션 트레이드오프

| 방식 | 장점 | 단점 |
|---|---|---|
| **(1) 매매법 라벨링** — 점수는 그대로, 카드에 "지금 셋업: 돌파/눌림/없음" attribute 만 추가 | 변경 작음, 데이터 누적 가능 | 시그널 분리 효과 약함, 사용자가 어차피 헷갈림 |
| **(2) 매매법별 Buy.Score / Exit.Triggers 분리** — `Buy.Score.Breakout` / `Buy.Score.Pullback` 별도 점수 + 매매법별 청산 트리거 | 본질 해결, 보유시간 분리 가능 | 구현 큼, 동시 발화 시 우선순위 룰 필요, 백테스트 필수 |
| **(3) 사용자가 모니터링 시작 시 매매법 선택** — `/on breakout` / `/on pullback` 모드 분리 | UI 명확, 사용자 의도 반영 | 한 종목이 두 셋업 다 통과하는 경우 처리 필요 |

## 5. 미해결 질문 (내일 이어서 답할 것)

1. **사용자가 평소 끌리는 매매법이 둘 중 명확한 게 있어? 아니면 둘 다 하고 싶어?**
   - 5/19 메이슨캐피탈 (+5.70%), 5/20 주성엔지니어링 3차 (+7.59%) 같은 빠른 익절 패턴 = 눌림 가까움
   - 본인 의도가 돌파 추종이었는지 / 눌림 익절이었는지 확인 필요
2. **위 순서대로 (a) 통설 자료 수집 → (b) 라벨링 백테스트 부터 시작할까, 아니면 사용자가 매매법 직접 정하고 그쪽 점수만 먼저 만들어볼까?**
   - Top-down: 통설 → 라벨링 → 검증 → 분리 도입 (정공법)
   - Bottom-up: 사용자가 한 매매법만 선택 → 그쪽 점수/청산만 먼저 구현 → 운영 검증 (빠른 시작)
