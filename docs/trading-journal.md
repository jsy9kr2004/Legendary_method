# 매매일지 작성 가이드

이 문서는 **Claude Code (또는 다른 AI 어시스턴트)** 가 사용자의 매매일지를 작성할 때
따라야 할 가이드다. 단순 레포트 자동화가 아니라 매번 다른 매매에 대해 **사후 평가 +
다각도 튜닝 포인트 도출**까지 하는 것이 목적.

## 핵심 의도 (사용자 비전, 2026-05-18)

> "감을 시스템으로 잡고 싶다. 그러러면 결국 매직 넘버를 데이터로 찾아가야 한다.
> 매매일지를 통해 (1) 그날 매매의 정직한 평가, (2) Buy.Score 가중치/Exit.Triggers 청산 임계의
> 튜닝 후보 도출, (3) 종목별 운전수 가설 검증 데이터를 누적하고 싶다."

→ 매매일지는 **단기 평가 + 장기 패턴 수집** 의 핵심 산물. 사용자의 감 + 시스템
시그널 + 결과의 3 axis 를 객관화한다.

---

## 0. 사용자 매매 룰 + 매매일지 작성 원칙 (2026-05-20 명시) ★

### 0.1 사용자 매매 룰 (Claude 는 baseline 으로 가정)

1. **매수**: Buy.Score 점수 🟢 **STRONG (≥5.0)** 일 때만 매수.
2. **매도**: Exit.Triggers 청산 시그널 (A1~A5 손절 / B1~B3 익절 / C1~C5 시그널) 중
   **하나라도 발화** OR **매수가 대비 -2% 도달** 시 매도.

→ 사용자는 위 두 룰을 일관되게 지키려고 노력함. 매매일지에서 "사용자가 시그널
무시" / "결정 일관성 부족" / "룰 위반" 같은 평가를 내리기 **전에**, 시간차 + 시그널
변동을 먼저 의심.

### 0.2 버튼-실거래 시간차 (반드시 고려) ★

사용자 동작 순서:
```
시그널 보고 (tick A 시점) → 주식창 전환 → 매수/매도 클릭 →
체결 → 봇에 /buy 또는 /sell 명령 입력 (trades 의 ts)
```

즉 **trades.ts 는 tick A 보다 수초~수십초 늦음** (`사용자 인지 lag`).

따라서:
- 매수 ts 의 closest tick 만 보면 사용자가 본 시그널과 다를 수 있음.
- **반드시 매매 ts 의 30초 전 윈도우** `[ts - 30s, ts]` 의 Buy.Score 점수/등급 변동과
  Exit.Triggers 트리거 발화 history 를 함께 본다.
- 매수: 윈도우 내 max(buy_grade) 가 STRONG 이었으면 사용자가 본 시그널이 STRONG.
- 매도: 윈도우 내 어느 시점 trigger 발화 history (Exit.Triggers A/B/C 어느 트리거?) 또는
  매수가 대비 -2% 도달 여부 함께 확인.

**금지 표현 (매매 ts 시점 시그널만 보고 결론 내리지 X)**:
- "시그널 NEUTRAL 인데 사용자가 매수" — 30초 전 STRONG 이었을 수 있음.
- "시그널 WATCH 유지인데 사용자가 빠르게 매도" — 30초 전 C 트리거 발화했을 수 있음.
- "사용자 결정 일관성 부족" — 시간차 + 시그널 변동 + 룰 준수 가정 먼저 검토.

### 0.3 시스템 튜닝 발전 원칙 ★

사용자는 시스템이 점진적으로 발전하길 원함. 허용 범위:
- 새 지표 추가
- 점수 계산 방식 변경
- 과도하다고 판단되는 지표 제거

단 **오버핏 방지 ritual** 통과 필수:

1. **한국 단타 통설 검증** — 변경 제안 시 "한국증시 주도주 매매법" 등 키워드로
   웹 검색 (WebSearch / WebFetch). **수익률이 검증되거나 통설로 회자되는 기법** 인지
   확인. 검색 결과 없는 자작 가중합/공식은 금지 (CLAUDE.md "검증 안 된 자작 가중합
   X" 원칙 강화).
2. **데이터 검증** — 현재 확보된 데이터 (`data/tick_logs/*.parquet`,
   `data/trades/*.parquet`) 로 변경 전후 백테스트/시뮬. **실제 수익률 또는 승률이
   올라가는지 확인**. 안 올라가면 변경 보류.

매매일지의 모든 튜닝 후보 가설은 위 두 단계를 **명시적으로 적어둬야** ritual 진행
가능. 1건 표본 단독 변경 영구 X.

→ 자세한 정책은 메모리 [[user-trading-rules]] / [[system-tuning-ritual]] /
[[user-trading-vision]] 참조.

## 2. 입력 데이터

### 2.1 사용자 제공 (필수)

- **차트 이미지** — HTS 캡쳐 (분봉/일봉/호가). Claude 의 멀티모달 입력으로 받는다.
- **일자** — `YYYY-MM-DD` 또는 "오늘"/"어제" 같은 자연어.

### 2.2 사용자 제공 (선택)

- 사용자 메모 / 감정 ("매도 시그널 떴는데 감으로 버텼다", "이상하게 손이 떨렸다")
- 외부 컨텍스트 (장 상황, 뉴스, 컨디션)

### 2.3 시스템 데이터 (Claude 가 쿼리)

| 경로 | 형식 | 내용 |
|---|---|---|
| `data/tick_logs/YYYY-MM-DD.parquet` | parquet, 40+ 컬럼 | 3초 tick 모든 시그널 + Buy.Score 점수 + Exit.Triggers 트리거 |
| `data/tick_logs/raw/YYYY-MM-DD.jsonl` | jsonl fallback | parquet 변환 전 raw |
| `data/trades/YYYY-MM-DD.parquet` (or jsonl) | parquet | `/buy` `/sell` 이벤트 + `trigger_fired` |
| `data/daily/ohlcv.parquet` | parquet | 다음날 갭/결과 매칭용 |
| `data/holdings.json` | json | 보유 종목 + 손익률 |

### 2.4 분석 도구

```
python -m src.analysis.replay CODE DATE        # 종목별 시계열 + Buy.Score breakdown
python -m src.analysis.regret DATE             # 그날 STRONG/WATCH + 매수 매칭
```

직접 pandas 쿼리도 OK — Claude 가 상황에 맞게 선택.

## 3. 분석 워크플로우 (Claude 가 따라가는 순서)

### Step 1. 데이터 수집

1. `data/tick_logs/{date}.parquet` 존재 확인. 없으면 raw jsonl 시도. 둘 다 없으면
   사용자에게 알림 + 가용 일자 list.
2. `data/trades/{date}.parquet` 로 매수/매도 이벤트 확인.
3. 매수한 종목 list 추출 → 각 종목별 `replay` 호출 결과 + 직접 parquet 쿼리.

### Step 2. 매수/매도 결정 시점 분석 (시간차 윈도우 포함) ★

**§0.2 시간차 원칙 필수 적용**. 매수/매도 ts 의 closest tick 단독으로 보지 X.

매수 시점의 시그널 분석:
- **매수 ts 의 [-30s, +5s] 윈도우** 의 tick row 들 전부 수집.
- **윈도우 max(buy_grade)** = 사용자가 클릭 전에 본 가능성 있는 최강 시그널.
  windowed max 가 STRONG 이었으면 룰 준수 진입.
- 클로즈스트 tick (매수 ts ± 3s) 의 시그널은 **클릭 후 실제 체결 시점** 시그널 —
  사용자 결정 근거가 아니라 결과 시점 시그널.
- buy_reasons (어떤 항목이 양수 가산), vol_accel, VP, candle, VWAP·MA — 윈도우 내
  변동 함께 표시.

매도 시점도 동일 — sell ts 의 [-30s, +5s] 윈도우:
- **윈도우 내 Exit.Triggers 트리거 발화 history** (A1~A5 / B1~B3 / C1~C5 어느 트리거가 언제
  발화?) — 사용자 룰 (b) 청산 시그널 매도의 근거.
- **매수가 대비 -2% 도달 여부** — 사용자 룰 (c) 손절 매도의 근거. tick 의 price
  추적.
- closest tick 의 buy_grade 가 STRONG/WATCH 유지여도, 윈도우 내 어느 시점에
  C 트리거 한 번이라도 발화했으면 사용자가 그 시그널을 보고 매도한 정상 케이스.

**금지**: "시그널 X 인데 사용자가 Y" 단정 결론. 항상 윈도우 내 시그널 변동을 먼저
표시하고, 변동이 사용자 룰과 정합한지 확인.

### Step 3. 보유 기간 동안 시그널 추이

`replay CODE DATE --since {매수시각} --until {매도시각}`:
- Buy.Score 점수 시계열 (높은 채로 유지? 떨어진 지점?)
- Exit.Triggers C1~C5 트리거 발화 history (얼마나 자주? 사용자 매도 시각과 시차?)
- 가격 vs MA5/MA20/VWAP 위치
- 외인/기관/프로그램 Δ 패턴 (round 36) — 매수 동반 vs 매도 동반

### Step 4. 사용자 차트 vs 시스템 시그널 cross-check

차트 이미지에서 보이는 것:
- 분봉 패턴 (양봉/음봉, 윗꼬리 크기, 거래량 봉)
- 사용자가 직접 표시한 매수/매도 지점
- 호가창 / 거래원 정보 (있다면)

이걸 시스템의 Buy.Candle candle / 매수가 비교. 일치하지 않는 부분은 시스템 데이터 결함
또는 사용자 시점 차이.

### Step 5. 다음과 비교 (가능하면)

- `regret DATE` — 그날 surface 됐는데 안 산 STRONG 종목 vs 사용자가 산 종목
- 다음날 시가 갭 (data/daily/ohlcv 의 다음 거래일 row) — 결과 확정
- 같은 종목의 과거 매매 데이터 (data/journal/ 또는 trades parquet 이력)

## 4. 매매일지 출력 형식 (md)

매매일지 1건 (사용자 검토 + data/journal/YYYY-MM-DD.md 로 저장 권고).

```markdown
# 매매일지 — YYYY-MM-DD (요일)

## 개요

| 종목 | 코드 | 진입 | 청산 | 보유 | 손익률 | 결과 |
|---|---|---|---|---|---|---|
| (종목명) | (6자리) | (HH:MM @가격) | (HH:MM @가격) | (분) | ±X.X% | 익절/손절/시간/감 |

(여러 종목이면 행 추가)

장 전반: 시장 국면 한 줄 (KOSPI/KOSDAQ 등락, 강세/약세, 거래대금 분위기).

## 종목별 분석

### 1. (종목명) (코드)

**매수 결정**
- 시각: HH:MM:SS @ ₩N (Buy.Score 등급/점수, 등급 도달 시각)
- 시그널 강도: 회전율 N위 / vol_accel 5m=X.X 1m=X.X / VP=N / candle=type
- 사유 (buy_reasons 상위 3): 항목별 가산점
- 같은 시간대 다른 후보: 어떤 종목이 더 높은 등급이었나?

**보유 중 시그널 추이**
- Buy.Score 점수 변동 (max 도달 시각, 최저 시각)
- Exit.Triggers 트리거 발화 timeline:
  - HH:MM:SS C2 Bearish Divergence (가격 +X / VP -Y)
  - HH:MM:SS C1 VP_5MA < 100 (현재 N)
  - ...
- 외인/기관 Δ 시점 (round 36) — 자금 흐름 방향 변화

**청산 결정**
- 시각/가격/방식 (Exit.Triggers 자동 트리거 vs 사용자 감)
- 발화한 트리거: A1/A2/.../C1~C5 (trigger_fired 컬럼)
- 트리거 발화 → 사용자 매도까지 시차 (N초/분)
- 만약 시그널 무시한 경우: 사용자 메모

**결과 평가**
- 손익률 vs 가설:
  - 이상적 청산 시점 (보유 기간 중 최고가) 대비 얼마나 손실?
  - Exit.Triggers 첫 트리거 발화 시점에 청산했다면 결과?
- 다음날 시가 갭 (있는 경우): 보유 유지했다면?

## 시스템 시그널 vs 사용자 감

- 매수: 시그널이 STRONG 이었고 사용자도 매수 → 일치
- 매수: 시그널 STRONG 인데 사용자가 안 함 → 불일치 (사유 분석)
- 매수: 시그널 NEUTRAL/WATCH 인데 사용자가 매수 → 사용자 감 우위? 또는 false negative?
- 매도: Exit.Triggers 트리거 발화 + 사용자 매도 = 일치 / 시그널 무시 + 사용자 매도 = 감 우위
- 매도: 트리거 발화 없는데 사용자가 매도 = 사용자가 본 시그널이 시스템에 X (튜닝 후보)

## 튜닝 포인트

### 단기 (이번 매매만)

Buy.Score 가중치 후보:
- (예) "Buy.VP VP_STRONG +2 가 매수 결정에 큰 비중. 이번 매매에선 VP 110 임계 직전
  108 에서 사용자가 매수 — 임계를 100 으로 낮추면 어땠을까?"

Exit.Triggers 청산 임계 후보:
- (예) "C2 Bearish Divergence 발화했지만 사용자가 버텨서 +3% 익절 도달. C2 단독은
  너무 빠른 청산 → 가중 임계 (vp_5ma_delta 절댓값 50 이상으로) 도입 검토."

### 장기 (현재 + 과거 누적)

- 같은 종목의 과거 매매 패턴 (data/journal/ 누적 검색):
  - "이 종목 은 항상 09:30 전 갭상 형성 후 10:00 전 청산 패턴 — 운전수 가설"
  - "이 종목 음봉 윗꼬리 후에도 회복하는 빈도 X%" → 종목별 Buy.Candle 가중치 약화
- 다른 비슷한 종목과의 비교 (예: 같은 테마, 같은 시총대)
- 1주/1개월 누적된 매매 결정 vs 결과 분포

### 운전수 가설 시그니처 후보

`docs/scalping-strategy.md` 의 Theme.Leader 외에 종목별 시그니처:
- 갭상 빈도 (그날 분봉 첫 5분의 양봉률)
- +30% 도달 시각 분포 (이른 종목 vs 늦은 종목)
- 분봉 자기상관 (운전수 있는 종목은 분봉 패턴이 연속적)
- 거래대금 spike 후 회복 시간

→ 이 시그니처들이 일관되게 나오면 Phase 3 종목별 파라미터 DB 의 marker 로.

## 후속 토론 / 피드백 (YYYY-MM-DD 시점 대화)

매매일지 작성 후 사용자 ↔ Claude 대화에서 나온 추가 분석을 일지에 함께 박는다.
**누적성** — 같은 가설이 5~10건 일지에 반복되면 ritual 통과 후보. 매매일지 단독은
일회성 기록이지만 후속 토론까지 누적되면 매직 넘버 튜닝 데이터셋이 된다.

### 토론 정리에 포함할 것

- 사용자 인지 vs 시스템 측정 (lag / 시그널 / 결과 차이) 정량 매핑
- What-if 시뮬 (시스템 시그널 그대로 따랐을 때 손익) — 표본 / 룰 / 결과
- 매매 안 한 STRONG 종목 list — 후회 후보
- 가설 → 검증 결과 (특히 **틀린 가설** 도 박을 것 — 다음에 같은 가설 재제시 회피)
- 사용자 직관 vs 정량 데이터 일치/불일치
- 5/19, 5/20… 누적할 데이터 항목

### 톤 원칙 (이 섹션도 §5 따름)

- 가설은 "이 1건 기준" 명시 — 표본 부족 인정
- 틀린 가설은 그대로 보존 (제거 X) — 누적 시 패턴 안 보임 방지
- 사용자 input (메모 / 답변) 은 별도 인용 또는 ★ 마커
```

### 4.1 형식 변형 — 매수 안 한 날

매수/매도가 없는 날도 매매일지 가치 있음:

```markdown
# 매매일지 — YYYY-MM-DD (요일) — 무거래

## 그날 surface 된 STRONG 종목 (regret 결과)

- (n=N) — 이 중 어느 것도 사용자가 매수 안 함

## 매수 누락 검토 (사용자 감 X vs 시스템 시그널 ◯)

각 STRONG 종목에 대해:
- 왜 안 샀나? (사용자 메모 — "장 분위기 안 좋아 보였다", "다른 일 했다" 등)
- 결과 (다음날 시가 갭) — 안 사길 잘했나? 후회되나?

## 장기 패턴

매수 안 한 날의 빈도 + STRONG 누락 종목의 결과 분포.
```

## 5. Claude 의 톤 가이드

매매일지는 **정직한 사후 평가**가 본질이다. 다음 톤 원칙:

1. **사용자는 §0.1 룰을 일관되게 지킨다고 가정** ★ — "시그널 무시 진입" / "결정
   일관성 부족" / "룰 위반" 단정 X. 그렇게 보이면 §0.2 시간차 윈도우 + 시그널 변동
   먼저 확인. 윈도우 내 STRONG / 트리거 발화 흔적 있으면 룰 준수 진입.
2. **사용자 결정 평가가 본질 X** — 알파의 본질은 **시스템 시그널 자체의 정확도**.
   매매일지는 "사용자 잘했나/못했나" 가 아니라 "시그널이 사용자가 본 시점에 어땠는지
   + 결과 + 튜닝 가설" 이 핵심.
3. **결과가 좋아도 결정 과정이 운이면 그대로 지적** — outlier 1건이 손익 좌우하는
   날은 "운 비중 높음" 명시. 단 사용자 행동에 대한 비판이 아니라 **시그널 정확도가
   outlier 의존인지 시스템적 알파인지** 분석.
4. **튜닝 후보는 항상 가설 + ritual 명시** — §0.3 ritual (통설 검증 + 데이터 검증)
   진행 필요성을 매번 명시. 1건 표본으로 가중치 변경 X. "이 가중치를 낮추는 게
   정답" 단정 표현 X.
5. **모순되는 신호 발견 시 모두 노출** — Buy.Score STRONG 인데 Exit.Triggers C2 Bearish 발화 같은
   복합 케이스에서 "어느 쪽이 옳다" 단정 X, 양쪽 다 평가.
6. **차트와 시스템 데이터 불일치는 시스템 데이터 결함 의심** — 사용자 차트가 양봉
   인데 Buy.Candle candle="bearish" 면 시스템 결함 가능성 우선 검토 (KIS 응답 / 시각 정렬
   문제 등).
7. **간결 + 정량적** — "느낌상" 같은 표현 X. 모든 평가에 시각/숫자 첨부.

## 6. 누적된 매매일지 활용

`data/journal/YYYY-MM-DD.md` 로 저장하면 다음 효과:

- **다음 매매일지 작성 시 과거 패턴 참조** — 같은 종목 매매 이력 / 비슷한 케이스
- **튜닝 후보 누적** — 여러 매매일지에서 같은 가중치/임계 조정 후보가 반복 등장하면
  ritual 통과 후 정식 변경
- **운전수 가설 시그니처 추출** — 종목별 패턴이 매매일지 N건에 일관되게 나오면
  Phase 3 파라미터 DB 의 marker 후보

매매일지 = Phase 2 의 가장 직접적인 산물. 사용자가 검토 + 보완하면 그 자체가 Phase
3 데이터셋이 된다.
